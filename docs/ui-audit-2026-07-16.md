# UI Audit — 2026-07-16

Triggered by live-app screenshots (Cards, Decks grid, Deck detail, Collection, Storage). Every finding below is verified against current code, not assumed from the screenshots alone.

## Summary

9 issues investigated. 7 are real, code-confirmed bugs/inconsistencies. 1 ("Storage locations not loading") turned out not to be a bug — see below. 1 (button-size inconsistency) is a systemic pattern, not a single fix.

## 1. Viewport width is inconsistent — confirmed, worst offender

`decks/[id]/page.tsx` alone uses **five different max-widths** in one page: tab bar `max-w-[1280px]`, Cards tab `max-w-[1080px]`, Analysis tab `max-w-[1080px]`, Combos tab unconstrained (no wrapper), Upgrade tab unconstrained (no wrapper), Strategy tab `max-w-[720px]`, plus the loading skeleton uses yet another value (`max-w-[960px]`/`max-w-[1280px]`).

Separately, the rest of the app splits into two camps: Decks grid, Collection, Storage, Shared Cards all use `max-w-[1520px]` (via `PageHeader`-based layout); Settings uses `max-w-[1280px]`. So there isn't one inconsistency, there are two: the deck-detail page disagreeing with itself, and Settings disagreeing with everything else.

**Fix:** pick one canonical content width (recommend `1280px` — matches the tab-bar width Gene already settled on, and reads better on ultrawide than 1520 for text-heavy tabs), add it as a token (`--content-max-width`), and apply it uniformly, including inside every deck-detail tab.

## 2. Header consistency — confirmed

`PageHeader.tsx` (used by Decks, Collection, Storage, Settings) correctly uses `text-[length:var(--fs-3xl)]` for its title. `PersistentHeader.tsx` (used only by deck-detail) hardcodes `text-[16px]` instead — bypasses the token system and renders visibly smaller/different from every other page's header.

`PersistentHeader.tsx` also hardcodes raw colors instead of the tokens already defined in `tokens.css`: `#EF9F27` / `rgba(239,159,39,0.15)` for the Precon-mod badge (should be `var(--signal-warning)` / `var(--signal-warning-bg)`), and `#1D9E75` / `rgba(29,158,117,...)` for the Post-game debrief button (should be `var(--accent-primary)` / `var(--accent-primary-bg)`).

**Fix:** swap the hardcoded size and colors in `PersistentHeader.tsx` for the existing tokens. No new tokens needed — they already exist and are unused here.

## 3. Button sizes are inconsistent — confirmed, systemic

`components/ui/button.tsx` is well-built: a proper `cva` scale (`xs`/`sm`/`default`/`lg`/`icon`/`icon-xs`/`icon-sm`/`icon-lg`) with consistent height, padding, and radius per size. The problem isn't the component — it's that **58 files** in `src/` use raw `<button>` elements instead of importing it, each hand-rolling its own padding/height. Concrete example: Storage's "Manage locations" button (`app/storage/page.tsx` line 97) is a raw `<button>` with `p-4` and no relation to any size token, sitting in the same grid as tiles that use the shared button scale elsewhere in the app.

**Fix:** this isn't a single-file fix — it's a sweep. Recommend a punch-list pass through the 58 files, replacing raw `<button>` with `<Button variant=... size=...>` where the element is actually a button (some are legitimately unstyled click targets like the ColorPicker swatch and don't need it). Worth doing as its own Gene phase, not folded into this one.

## 4. "Synced" label — confirmed, dead

`app/collection/page.tsx` lines 211–219:

```tsx
<span className="flex items-center gap-1.5 text-[length:var(--fs-xs)] text-[var(--text-tertiary)]">
  <RefreshCw className="size-3" aria-hidden="true" />
  Synced
</span>
```

Hardcoded string, not wired to any real sync timestamp or status. Two options: delete it outright, or wire it to a real last-import timestamp if that's a data point you want visible. Given there's no "sync" concept left in the architecture (collection is reimport-based, not live-synced), I'd delete it.

## 5. Storage locations "not loading" — investigated, not a bug

I checked the full path: `storage_locations` table schema (has `sort_order`, defaults to 0 — the `.order('sort_order')` call in the overview route is safe), the `/api/storage/overview` route (correctly counts `physical_copies` per location), and `app/storage/page.tsx`'s render logic (correctly renders whatever the route returns). No code defect anywhere in this chain.

Then I checked where `storage_location_id` actually gets set, and found the real explanation: **`warm-start-import.ts` (the Archidekt import path) never sets `storage_location_id` on any inserted `physical_copies` row.** Every card that comes in through import lands with `storage_location_id = NULL` — i.e. Unsorted — by default. `storage_location_id` only ever gets set afterward, one copy at a time, through `LocationPickerModal` / the `assign-location` PATCH route.

[Likely] Trade Binder and Bulk Box showing 0 cards is correct, not broken — you created those locations but nothing has been manually assigned to them yet, which is consistent with Unsorted holding all 821 cards. If there's a separate visual glitch (like a stuck spinner near the Unsorted tile) that's a different, purely client-side issue I can't diagnose from code alone — if it's still happening, a screenshot with the browser dev tools network tab open would pin it down fast.

## 6. Deck-tile post-game shortcut — confirmed, scoped

`DeckTile.tsx` lines 94–133: hovering the commander art reveals a two-button overlay — "Post-game" (`/decks/${id}?debrief=true`) and "Open" (`/decks/${id}`). Only the text/info section below the art (lines 137–148) is a plain single-click link to the deck. So the card as a whole doesn't go straight to the deck profile — the art area requires a second click on "Open."

Confirmed this pattern is scoped to `DeckTile.tsx` only — `DraftDeckTile.tsx` (used for in-progress brews) doesn't have this overlay at all, it uses a single `handleContinueBrewing` click handler.

**Fix:** remove the hover overlay entirely, make the whole tile (art + info) a single link to `/decks/${id}`. The debrief flow should live inside the deck profile (it already has a "Post-game debrief" button in `PersistentHeader.tsx`, so nothing is lost — just removing a redundant, click-blocking entry point from the grid).

## 7. Storage doesn't function like Decks — confirmed, functional gap

Structurally the two hub pages are already reasonably aligned (`PageHeader`, `max-w-[1520px]`, grid of tiles). The gap is in the detail view. Deck detail (`decks/[id]/page.tsx`) has 5 tabs (Cards/Analysis/Combos/Upgrade/Strategy), a grid/list view toggle for cards, search, and grouping. Storage detail (`storage/[locationId]/page.tsx`) is a single flat list — no grouping, no grid/list toggle, no search/filter, no tabs. It works, but it's a much thinner experience for what's conceptually the same kind of "container full of cards" view.

**Fix, scoped:** at minimum bring over the grid/list toggle and search that `CardsTab` already has — that's the highest-value, lowest-effort parity gain. Full tab parity (Analysis-equivalent for a storage location, e.g. isn't really meaningful) isn't warranted; recommend not overbuilding this one.

## 8. Design tokens — baseline established, no new issues beyond what's above

`tokens.css` is in decent shape: 8pt spacing scale, a typographic scale (`--fs-xs` through `--fs-3xl`), status/WUBRG/layout tokens, plus a documented "legacy alias" block bridging to shadcn variable names. The token system itself isn't the problem — the problem (items 2, 3, 6) is components not using it. No action needed here beyond fixing the call sites already listed above.

## 9. Blurred commander background for deck profile — scoped, ready to build

`decks/[id]/page.tsx` already has `deck.commander_scryfall_id` available (it's fetched into the page's deck object, line 28). The Scryfall image URL pattern is already used elsewhere in the codebase (`storage/[locationId]/page.tsx`): `https://cards.scryfall.io/{size}/front/{c1}/{c2}/{id}.jpg`. For a background, `art_crop` is the right size (wide aspect, no card frame).

**Implementation shape:** a `fixed`/`absolute`-positioned div behind `PersistentHeader` + tab content, `background-image` set to the art-crop URL, `filter: blur(...)`, low opacity, `z-index` below all real content. Scope it inside `decks/[id]/page.tsx` only (not `layout.tsx`), so it doesn't leak onto other pages. Straightforward, no new data plumbing needed.
